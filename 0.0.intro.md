### 🚀 [Oh My Cap] 프로젝트 전체 수행 절차 (SDLC)

#### ✅ Phase 1. 스펙 및 요구사항 정의 (Planning & UI Design) - `[완료]`
우리가 무엇을 만들지, 어떤 모습일지, 어떻게 조작할지를 결정하는 가장 중요한 단계입니다.
1. **PRD 작성 (완료):** 망분리 환경, 페르소나 도입, 모드(Plan/Act), 성능 지표 등 핵심 비즈니스 로직 확정.
2. **UI 와이어프레임 설계 (완료):** 설정 패널 5개 탭, Tool Window 채팅창, 우클릭 메뉴의 인터페이스 도면 완성.
3. **시나리오 시각화 (완료):** 사용자가 플러그인을 어떻게 쓸지 스토리 기반 Mockup 이미지 구현.

---

#### 📍 Phase 2. 시스템 아키텍처 및 인터페이스 설계 (Architecture & API Spec) - `[현재 진행할 단계]`
그림으로 그려진 플러그인이 **"사내 LLM 서버와 어떻게 말을 주고받을 것인가?"**를 기술적으로 정의합니다.
1. **시스템 파이프라인(Architecture) 설계:** IntelliJ 플러그인 ↔ LangGraph 에이전트 ↔ RAG DB 컴포넌트 간 데이터 흐름도를 다이어그램(Mermaid 등)으로 시각화.
2. **REST API 및 Tool Calling 규격 정의:** OpenAI 호환 스트리밍 포맷과 플러그인 터미널/에디터 제어를 위한 JSON 스키마 명세서(API Spec) 작성.
3. **IntelliJ 컴포넌트 파악:** 소스코드를 읽기 위해 필수적인 IntelliJ 내부 OpenAPI(PSI 파서, Action System) 사용 규칙 정의.

---

#### 🛠️ Phase 3. 백엔드(서버) 코어 개발 (Backend / Agent Logic)
실제 뇌 구조 역할을 할 DGX 서버 측의 파이썬 기반 에이전트 시스템을 코딩합니다.
1. **LangGraph 상태 머신 구축:** 단순 질문인지 다중 파일 수정인지 판단하는 라우팅 엔진, 자가 검증(Critique) 오류정정 루프 등 랭그래프 뼈대 구현.
2. **검색(RAG) 파이프라인 실장:** 연관 Mapper, Entity, JSP를 물고 올 내부 VectorDB 스키마 구성 및 검색 모듈 개발.
3. **가상 도구(Tools) 모듈화:** 코드를 PMD 규칙으로 스캔하거나 SQL을 포매팅·검증해주는 단일 기능(Skills)들의 스크립트 모음 컴포넌트화.

---

#### 💻 Phase 4. 프론트엔드(플러그인) 클라이언트 개발 (IntelliJ UI/UX)
자바/코틀린을 사용하여 빈 프레임(IntelliJ)에 와이어프레임에서 잡은 디자인 요소들을 실장합니다.
1. **플러그인 껍데기 셋업:** Gradle 기반 플러그인 프로젝트 생성.
2. **UI 렌더링:** Tool Window의 채팅 뷰 컴포넌트, `[Diff / Apply]` 코드 비교 UI 창, 5개의 `[Settings]` 패널 실제 코딩.
3. **데이터 바인딩 및 후킹:** 에디터에서 코드 드래그 시 텍스트 가로채기(Hooking), 입력값을 11434 포트 API 규격으로 전송 및 스트리밍 답변 파싱 적용.

---

#### ⚙️ Phase 5. E2E 운영 워크플로우 통합 (Workflow Automation)
순수 코딩 영역을 넘어 사내망 시스템과의 결속 및 "진짜 편한 유지보수"의 정점을 찍는 통합 단계입니다.
1. **외부 연동 (JIRA, SVN/Git):** 플러그인에 입력된 인증 토큰으로 JIRA 서버 REST API 통신 실장. Git Commit 창 단축 자동 렌더링.
2. **산출물 파이프라인 연계:** 플러그인이 내장형 Selenium으로 무두(Headless) 브라우저를 띄워 캡처하고 `/docs/omc-reports` 폴더에 마크다운과 사진을 떨구도록 I/O 기능 개발.

---

#### 🧪 Phase 6. 통합 QA 환경 테스트 및 릴리즈 (Testing & Release)
만들어진 `플러그인` + `백엔드` + `사내망(JIRA)` 인프라를 연결하여 통합 테스트를 진행합니다.
1. **보안/망분리 테스트:** 외부 사이트(텔레메트리 등) 정보 유출이나 트래픽 시도 없이 오로지 지정된 단일 내부 IP로만 통신하는지 검증.
2. **스트레스/OOM 테스트:** 수천 줄짜리 레거시 자바 코드를 통째로 던졌을 때 플러그인이 컨텍스트 압축 로직을 통해 뻗지 않는지 방어력 검증.
3. **오프라인 빌드 (최종):** 사내 배포망에 얹을 `omc-v1.0.zip` 파일을 패키징하고 사용자 가이드와 함께 공식 릴리즈(Sign-off).
