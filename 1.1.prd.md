# [PRD] 엔터프라이즈 폐쇄망 전용 Agentic AI 코딩 어시스턴트 플러그인 (Final v7.0)

## 1. 문서 개요
- **프로젝트명:** Oh My Cap (약칭: **omc**)
- **목적:** 폐쇄망(Air-gapped) 환경에서 소스코드 외부 유출 없이 작동하며, 개발자가 직접 통제할 수 있는 권한과 프롬프트 시스템을 바탕으로 레거시 소스코드 유지보수를 돕는 'IDE 통합형 에이전틱 AI 코딩 플러그인'.
- **주요 대상:** 기존 모놀리식 운영 환경 및 레거시를 다루는 백엔드/프론트엔드/모바일 개발자.

## 2. 배경 및 목적
- **폐쇄망 특화 보안:** 소스 코드와 사내망 데이터의 완전한 보호.
- **운영 중심의 E2E 문제 해결:** 새로운 코드를 구현하는 것을 넘어, 기존 시스템의 [요구사항 분석 -> 수정 -> 리뷰 -> 브라우저 캡쳐 테스트 -> 산출물 정리 및 커밋]에 이르는 E2E 운영 업무 플로우 전반을 자동화.
- **환각 억제 및 사용자 통제력 강화:** 막연한 RAG 봇 아키텍처를 탈피하여, 사용자가 에이전트의 역할(Persona)과 수정 권한, 도메인 스코프를 능동적이고 완벽하게 통제함.

## 3. 타겟 환경 및 시스템 
### 3-1. 유지보수 대상 (Target Codebase)
- **백엔드:** 커스텀 Spring 프레임워크 기반 Java 
- **프론트엔드:** HTML, CSS, JavaScript, jQuery, JSP 
- **DB 및 모바일:** Oracle DB, Native Android/iOS

### 3-2. IDE 호환성 및 에이전트 권한 제어
- **Target IDE:** IntelliJ IDEA (2021.2 버전 이상 권장).
- **Plan / Act 모드 토글 (에이전트 자율성):** 
  1. **Plan 모드:** 요구사항을 분석하여 다중 파일 수정 방안과 단계별 실행 계획(Strategy)을 먼저 수립하고, 구체적인 **TODO 리스트**를 작성하여 개발자에게 제시하는 기획/검토 모드.
  2. **Act 모드:** 수립된 Plan(TODO 리스트)을 바탕으로, 에이전트가 순차적으로 파일을 직접 수정하고 단위 테스트 구동, 빌드 및 스크립트 실행(Terminal 연동) 등 실질적인 워크플로우 액션을 자율 수행하는 실행 모드.

## 4. 백엔드 인프라 및 에이전트 서버 아키텍처 (Server-Side)
단순한 프롬프트 스크립트를 넘어, AI가 자율적으로 사고하고 절차를 밟을 수 있게 설계된 **서버단(Middleware/LLM Serving) 핵심 구조**입니다.

### 4-1. 하드웨어 및 코어 모델
- **서버 환경:** 사내망 독립 환경, NVIDIA DGX™ OS, Ubuntu Linux Docker 컨테이너
- **하드웨어 스펙:** NVIDIA® GB10 Grace Blackwell Superchip, 128GB LPDDR5x RAM, 1TB.
- **코어 모델 (자율 선택 및 확장 구조):** 특정 모델에 종속되지 않으며 모델 선택이 자유로움. Qwen3-Coder-30B-A3B를 최우선 최적화 기준으로 삼되, 사용자가 Base URL, 포트 번호(기본 11434) 등을 변경하여 유연하게 호환 운영함.

### 4-2. OpenAPI 프로토콜 규격 (API Compatibility)
- **REST API Streaming 호환성:** 플러그인(UI)과 서버의 결합도를 완벽히 분리. 범용적인 OpenAI API 표준 프로토콜(`/v1/chat/completions` 포맷)을 준수하도록 백엔드 프록시를 구축하여, 타 IDE 확장이나 차세대 오픈 모델 교체 시 개발 비용을 최소화함.
- **Tool Calling (Function Calling) 준수:** 에이전트가 플러그인의 터미널 제어, 브라우저 캡쳐, 파일 쓰기 권한을 원활히 대리 수행할 수 있도록 표준 JSON Schema 기반의 Tool Calling 규격을 구현함.

### 4-3. 랭그래프(LangGraph) 기반 프로세스 오케스트레이션
LLM의 단발성 응답(환각)을 방지하고, E2E 운영 유지보수 프로세스를 체계적으로 이행하기 위해 **랭그래프(LangGraph)** 프레임워크를 활용, 상태(State) 기반의 논리적 절차 회로를 서버 측에 설계합니다.
1. **의도 분석 라우터 (Intention Router):** 개발자 질문이 단순 코드 설명인지, 다중 파일 수정인지, 복잡한 자동화(워크플로우)인지 판별하여 다음 노드(Node) 흐름을 결정.
2. **영향도 RAG 주입 노드 (Context Retriever):** 수정 Task로 판별 시, 단일 파일을 넘어 연계된 Mapper/JSP/Entity의 논리적 스키마를 내부 VectorDB 등을 통해 검색(Fetch)하여 프롬프트에 강제 병합.
3. **코드 생성 및 자가 검증 루프 (Self-Correct & Critique Loop):** 
   - `작성(Draft) 노드`가 코드를 생성하면, `비평(Critique) 노드`가 보안 취약점(SQL Inject)이나 PMD 룰셋 위반 여부를 스스로 검열. 오탐/오답 시 수정 루프 실행.
4. **IDE 액션 호출 (Tool Execution):** 검증을 마친 결과물(Diff, Action 명령어)을 플러그인으로 전송한 뒤, IDE 터미널 테스트 결과(Success/Fail 로그)를 넘겨 받아 재수정 루프를 돌릴지 캡쳐 산출물 단계로 넘어갈지 판단하는 상태머신 사이클 제어.

## 5. 플러그인 UI 및 상세 인터랙션 명세 (Tool Window)
### 5-1. 메인 채팅 화면 (Tool Window)
- **헤더 영역:** `Oh My Cap` 로고 / `Persona 뱃지(역할)` / 설정 버튼.
- **대화 및 Task 관리 (History & New Chat):** 
  - **새 대화 (+ New Chat):** 누적된 기존 컨텍스트(토큰)를 완전히 초기화하고, 다른 로직 분석이나 독립적인 빈 작업(Task)을 시작하기 위한 버튼.
  - **히스토리 서랍 (History Panel):** 과거 진행했던 대화 스레드를 보관하는 리스트형 패널. 첫 질문 20자 기준으로 제목이 자동 기입되며, 과거 항목 클릭 시 당시 마운트된 컨텍스트 파일 기록과 대화 내용을 완벽히 복원함.
- **토큰 인디케이터:** 채팅 본문 최상단에 실시간 토큰(Token) 점유율 게이지 바 표시.
- **동적 대화 방:** 스트리밍 텍스트 렌더링, 브라우저 캡쳐, 도구 실행 로그, 코드 적용 카드 (Diff / Apply / Ignore).
- **입력 프롬프트 영역 (에이전틱 컨트롤 요소):** 
  - **Plan/Act 모드 토글 스위치:** 대화 순간마다 기획/실행 권한을 즉시 스위칭.
  - `@` 멘션 컨텍스트 마운팅 및 📎 첨부 버튼 (로컬 문서/소스코드 연동).
  - **Rules (규칙):** 사내 코딩 컨벤션, 보안 지침 등 커스텀 규칙(예: `.rules`)을 글로벌 컨텍스트로 에이전트에게 강제 적용하는 기능. 
  - **Workflows (워크플로우):** '신규 API 명세 -> Controller 작성 -> Mapper 생성' 처럼 여러 단계로 나열된 복잡한 절차적 작업을 일괄 지정하여 자동 수행 제어.
  - **Skills (특화 스킬):** 특정 사내 레거시 프레임워크나 시스템 쿼리 변환 등 에이전트용으로 튜닝된 커스텀 모듈/기술을 호출.
  - **Hooks (이벤트 훅):** 파일 저장(Save), 브랜치 이동, 커밋(Commit) 직전 등 IDE의 특정 개발 이벤트에 에이전트가 자동으로 트리거되어 검사를 이행하도록 연동.
  - 다중 롤백(`Undo`), 실시간 답변 중지(`Stop`), 전송 버튼.

### 5-2. 컨텍스트 메뉴
1. **Add Snippet to Chat (채팅으로 보내기):** 드래그로 블록 지정한 소스코드를 즉시 메인 채팅창(Tool Window)의 입력 컨텍스트로 전송하여 자유 대화 개시.
2. **Explain Code Semantics (코드 문맥 설명):** 단순한 구문 해석을 넘어 해당 코드블록의 역할과 동작 맥락(Context)을 요약.
3. **Review Code Quality (코드 품질 평가):** 잠재적 버그 위험(Bug-prone), 가독성, 유지보수성 항목을 전문적으로 리뷰.
4. **Refactor Code Snippet (코드 리팩토링 제안):** 클린 코드 원칙 및 성능 최적화가 적용된 구조적 개선안 제공.
5. **Inspect Static Analysis [PMD] (PMD 정적 규칙 검사):** 순환 복잡성을 측정하고, 룰베이스 기반의 잠재적 레거시 오류 탐지.
6. **Analyze Impact Scope (영향 범위 분석):** 코드 변경에 따른 상/하위 모듈(Entity-Mapper-JSP) 간의 종속성(Dependency) 및 파급 범위 추적.
7. **Generate Unit Tests (단위 테스트 생성):** 방어적 운영을 위한 엣지(Edge) 케이스 중심의 테스트 작성.
8. **Validate SQL Query (SQL 정합성 및 실행계획 검증):** 쿼리 구문(Syntax) 오류와 Full Scan 위험성을 분석하고, Update/Delete 시 영향받는 Row 범위를 경고.
9. **Scan Security Vulnerability (보안 취약점 스캔):** SQL Injection, XSS 패턴 검증 및 로그 객체 내 포함된 개인 식별 정보(PII) 평문 노출 여부 스캔.

### 5-3. 환경 설정 패널 (Settings - 5개 탭)
- **[LLM Server] (오픈 모델 제어 및 서버 연결):**
  - **다중 프로파일 통신망:** 사내망 Base URL 포트와 API Key를 저장하는 프로파일.
  - **모델 동적 선택 (Model Selection):** vLLM, Ollama 등 로컬 서버가 서빙 중인 현재 모델(Qwen3 등) 리스트를 호출하여 실시간으로 타겟 모델을 교체(Switching).
  - **파라미터 튜닝:** 창의성을 억제하는 Temperature(0.0~0.3 코딩 권장), Top-P 등 오퍼레이션 값 조정.
  - **컨텍스트 윈도우 크기 (Context Window Size):** GPU 메모리 여건에 따라 프롬프트 허용 토큰 상한선 지정 (예: 32K, 128K).
  - **컨텍스트 스마트 압축 (Context Compression):** 사용자가 대화창에 할당한 소스코드나 문서가 지정된 Context Window를 초과할 경우, 중요도 기반으로 문맥을 요약(Summarize)하거나 절삭(Truncate)하여 메모리 부하 에러를 방지하는 토글 옵션.
- **[MCP]:** 내부망 JIRA/Redmine 등 확장 연동 프로파일.
- **[Persona] (역할 및 시스템 프롬프트 제어):** `Requirements Analyst(요구사항 분석가)`, `Backend`, `Frontend`, `DB Admin` 등. 플러그인이 제공하는 고정된 역할뿐만 아니라, 사용자가 스스로 `[+ 추가]` 버튼을 눌러 프로젝트 상황에 맞는 페르소나를 언제든 무제한으로 신규 등록(동적 확장)하고 각각의 도메인 시스템 프롬프트를 커스텀할 수 있는 기능.
- **[Prompts]:** 우클릭 메뉴 액션 프롬프트 구문 커스텀 관리.
- **[General]:** VCS 기본 형태(Git/SVN), 산출물 저장 경로 지정.

## 6. 에이전틱 E2E 운영 유지보수 워크플로우
### Phase 1. 요구사항 분석 및 방향성 도출 (Analyze)
- **1-1. 수정 요청 문서 수령 (Multimodal Input):** 
  - *Option A:* 채팅창 하단 📎(첨부) 버튼을 통해 Word, PDF 등 문서를 직접 업로드.
  - *Option B:* [MCP] 탭 연동을 통해 사내 ITS(Jira 등) 티켓 번호로 직접 요구사항을 자동 수집(Fetch).
- **1-2. 인터랙티브 구체화:** 요약 내용을 기반으로 인터랙티브(Interactive) 핑퐁 대화를 통한 제약사항 및 사이드 이펙트 점검.
- **1-3. 수정 범위 도출:** 변경이 필요한 패키지, 데이터베이스 쿼리, JSP 뷰 목록 사전 산출 및 정의.

### Phase 2. 에이전트 수정 및 단위 테스트 (Modify & Test)
- **2-1. Agent 자율 수정:** 도출된 수정 범위를 바탕으로, 관련 계층 파일 동시 수정.
- **2-2. 단계별 승인 (Human-in-the-loop):** `Diff View` 카드를 통해 개발자가 단계별로 코드를 리뷰하고 `Apply` 승인 처리.
- **2-3. 단위 테스트 진행:** 승인 코드에 한정하여 JUnit 구동.

### Phase 3. E2E 웹 자동화 테스트 및 화면 캡쳐 (Selenium Capture)
- **3-1. 시나리오 지정:** 화면 변경의 정상 작동 확인용 시나리오 문서 주입 (URL 주소, 버튼 ID 지정).
- **3-2. 내장 Selenium 엔진 구동:** 에이전트가 시나리오에 따라 로컬 웹 화면을 대상으로 실제 클릭/입력 동작 제어.
- **3-3. 최종 스크린샷 캡쳐:** 변경된 JSP/웹 화면을 백그라운드 캡쳐하여 결과 보고서에 렌더링 준비.

### Phase 4. 최종 산출물 및 형상 적용 (Commit & Docs Export)
- **4-1. 산출물 일괄 자동 생성 (Custom Export Root):** 프로젝트 내 지정 폴더(예: `/docs/omc-reports`)에 최종 산출물 자동 생성 (*수정 자원 목록표*, *소스 파일 비교 문서 (Diff)*, *캡쳐 뷰 포함 테스트 보고서*).
- **4-2. 적응형 형상 관리 연동 (Git/SVN):** 지정된 VCS 방식에 맞춰 커밋 메시지 초안을 생성하고, 플러그인 상에서 SVN Commit 또는 Git Commit 창을 자동 활성화.

## 7. 프로젝트 성공 지표 (KPIs & Metrics)
이 플러그인의 도입 성공 여부를 판단하기 위해 수집하고 트래킹(Tracking)해야 할 핵심 지표입니다.
1. **코드 채택률 (Acceptance Rate):** AI가 제안한 소스코드(Diff View) 중 개발자가 실제로 `Apply(승인)`한 비율. (MVP 목표: 도입 초기 30% 초과)
2. **리드타임 단축률 (Time-to-Resolution):** 시스템 장애 또는 JIRA 티켓 접수부터 운영 커밋/보고서 배포까지의 평균 소요 시간 증감률.
3. **오류 롤백 수 (Undo Count):** 단위/화면 테스트 실패로 인해 개발자가 다중 롤백을 선언한 평균 횟수 (품질 및 환각 억제 측정 지표).

## 8. 제약사항 및 개발 제외 범위 (Out of Scope / Non-Goals)
프로젝트 스코프 팽창을 방지하기 위해 v1.0 MVP 구현 시 명확하게 제외되는 기능과 인프라 제약 조건입니다.
1. **개발 미지원 환경:** "VS Code, Eclipse 플러그인 등 타 IDE 환경은 배제하고 오직 IntelliJ Ultimate/Community 생태계 호환성에만 집중한다."
2. **CI/CD 및 운영 배포망 격리:** 개발 로컬 환경을 벗어난 "운영망(Production) 서버로의 직접 자동 배포/파이프라인 연계"는 보안 치명도를 고려해 이번 버전에선 원천 배제한다.
3. **인프라 병목 대기 채널 (Concurrency Queueing):** 수십 명의 전사 개발자가 동시 다발적으로 추론/영향도 분석을 호출해 DGX 서버 API(11434 포트)에 병목이 발생할 경우, 모델 OutOfMemory 방지를 1순위로 두어 플러그인 채팅창에 "대기 중(Queue)" 폴백 UI 노출을 의무화한다.
4. **플러그인 오프라인 배포 (Distribution):** 외망 연결을 요하는 IntelliJ Marketplace 배포가 불가능하므로 사내 포털망을 통해 `omc-v1.0.zip` 형태의 플러그인을 제공하며, 개발자는 "Install Plugin from Disk" 모드로 스탠드얼론(Standalone) 설치를 진행한다.

## 9. 마일스톤 및 릴리즈 로드맵 (Phased Roadmap)
에이전틱 E2E 워크플로우의 구상을 한 번에 개발하는 리스크를 줄이기 위한 3단계 점진적 배포 전략입니다.
- **Phase 1 (코어 UX 및 모델 통제 연동):** 채팅창 및 8대 우클릭 메뉴 액션, `Prompts/Persona` 탭 등 에이전트 통제 화면 구성, 단일 파일 단위 Diff/Apply 인터랙션 구현에 집중.
- **Phase 2 (영향도 분석 및 내부망 결속):** `Impact Analysis`를 활용한 다중 파일(Entity-Mapper-JSP) 동시 수정 뷰어, `[MCP]` 채널을 통한 사내 Jira 티켓 추적 연동, JUnit 테스트 코드 생성 오케스트레이션 구현.
- **Phase 3 (E2E 에이전트 완성형 배포):** Selenium 내장/후킹을 통한 로컬화면 시나리오 자동 제어 및 캡쳐 로직 구현, 최종 산출물(리포트, Diff 텍스트) 파일 자동 조립 및 SVN/Git 커밋 후킹 파이프라인 전면 배포.
