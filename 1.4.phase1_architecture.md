# [Architecture/Design] Oh My Cap MVP (Phase 1) 

## 1. 개요 (Overview)
본 설계 문서는 "Oh My Cap (omc)" 플러그인의 가장 핵심적이고 즉시 가치를 창출하는 **Phase 1 (최소 기능 제품, MVP)** 에 초점을 맞춥니다. 거대한 LangGraph 기반 자율 오케스트레이션과 Selenium UI 테스트 등 복잡도를 야기하는 기능을 일시 배제하고, 오직 **안전한 폐쇄망 환경에서 개발자가 LLM과 안정적으로 컨텍스트를 교환하고 코드를 수정(Diff & Apply)** 할 수 있는 코어 파이프라인만 구축합니다.

## 2. 시스템 아키텍처 (Core Components)

### 2.1 Plugin UI 계층 (IntelliJ Tool Window)
- **채팅 패널 (Chat Window):**
  - 스트리밍 응답 렌더링 지원.
  - 마크다운(Markdown) 구문 파싱 및 코드 블록 신택스 하이라이팅.
  - 마운트된 컨텍스트(현재 활성화된 파일, `@`로 첨부된 파일 등)를 볼 수 있는 뱃지 UI.
- **컨텍스트 액션 (Editor Context Menu):**
  - 드래그한 코드 영역 우클릭 메뉴 구현 (`Explain`, `Refactor`, `Unit Test`, `PMD Analysis`).
- **Diff & Apply 카드 (Action Card):**
  - 에이전트가 코드를 제안했을 때 즉시 원래 소스코드에 삽입하지 않고, 임시 Diff Viewer를 통해 좌/우(Original/Generated) 비교 제공.
  - 사용자가 `Apply` 버튼을 누르면 에디터의 Write Action을 통해 실제 편집기에 삽입/치환.

### 2.2 컨텍스트 매니저 (Context Manager)
- **통합 컨텍스트 파이프라인:**
  - `@` 멘션을 통한 프로젝트 내 특정 파일 및 폴더 로드.
  - 현재 개발자가 화면에 띄우고 있는 탭(Active Editor)의 소스코드 자동 주입.
  - 커스텀 Persona (예: Backend, Code Reviewer) 선택에 따른 Base System Prompt 결합.

### 2.3 LLM 게이트웨이 / 프록시 (REST Client Layer)
- **OpenAI 호환 어댑터 (`/v1/chat/completions` 포맷):**
  - 플러그인 컴포넌트를 LLM 제공자 라우팅과 완벽히 격리. URL과 포트, API Key, Model 이름만 Settings UI에서 받아 HTTP 클라이언트로 요청 전송.
  - (서버측) vLLM, Ollama 등 로컬 LLM이 OpenAI 포맷의 입출력을 준수하도록 통일 설정.
- **스트리밍 클라이언트 (SSE):** 
  - 긴 코드 작성 및 추론 시간 동안 사용자 대기(UX 병목)를 줄이기 위해 Server-Sent Events 기반의 Chunk 스트리밍 수신.

### 2.4 안정성 및 방어 로직 (Safety & Consistency)
- **에디터 상태 동기화 (Dirty State Check):**
  - 질문 발송 시점의 `Document` `ModificationStamp`를 기록하고, 스트리밍이 끝나고 `Apply`를 누르는 시점과 일치하는지 비교하는 로직 필수 구현. 불일치 시 충돌(Conflict) 경고 UI 노출.
- **OOM(Out Of Memory) 방어 (내부 Tokenizer):**
  - 서버로 HTTP 요청을 보내기 전, 플러그인 내부에서 `JTokkit` 등 경량 토크나이저를 사용해 Context 토큰 수를 계산.
  - 모델의 최대 Context Window(예: 32K)를 넘으면 사용자에게 경고하고 낮은 중요도의 파일부터 자동 Truncate(절삭)하여 400 Bad Request 및 서버 뻗음(Crash)을 사전 차단.

## 3. 워크플로우 명세 (The Wedge Workflow)

1. **Trigger:** 개발자가 코드를 블록 지정하여 "Refactor" 메뉴를 누르거나, 채팅창에 특정 파일(`<@User.java>`)을 멘션하여 리팩토링 요청.
2. **Context Build & Safety Check:** 프론트엔드가 페르소나, 파일, 질의를 묶고 내부 Tokenizer로 크기를 점검. 동시에 원본 문서의 `ModificationStamp`를 Snapshot 저장.
3. **Request:** 지정된 사내망 로컬 LLM 서버(VLLM 등) 주소로 스트리밍 API 호출.
4. **Stream & Render (Graceful Fallback):** 반환 데이터를 실시간 렌더링. 코드 블록(```)이 불완전하게 오더라도 UI가 크래시나지 않도록 방어하는 Parser 가동, 이후 `Diff/Apply` 카드 생성.
5. **Human-in-the-loop & Apply:** 개발자가 `Apply` 클릭 시, 플러그인은 현재 에디터의 `ModificationStamp`가 질문 당시(2번 단계)와 동일한지 확인 후 상태가 깨끗할 때만 변경사항을 편집기에 안전하게 반영.

## 4. 향후 로드맵 (Out of Scope for MVP)
- 다중 파일 동시 수정 뷰 및 `Plan` 모드에 따른 백엔드 LangGraph 오케스트레이션.
- Jira / Redmine 등 ITS 연동.
- 터미널 자동 실행(스크립트 트리거) 및 로컬 브라우저 캡쳐 자동화.
